# 2-Pager 템플릿 (Executive Summary)

## 개요
이 문서는 제품 요구사항의 핵심을 2페이지로 요약한 문서입니다. 경영진, 이해관계자, 또는 빠른 의사결정이 필요한 상황에서 사용합니다.

**⚠️ 중요: 2-Pager 작성 전 반드시 확인하세요**
- PRD 작성 가이드라인: `docs/Template/PRD_GROUND_RULES.md`를 먼저 읽고 준수하세요.
- 핵심 원칙: **Logic over Tech** - 기술적 구현 상세가 아닌 비즈니스 로직과 정책 중심으로 작성합니다.

---

## 문서 구조

### 1. 문서 정보
**목적**: 작성자, 작성일, 최종 수정일, 버전(Version)

**작성 가이드**:
- 문서 작성자 및 담당자 정보
- 작성일 및 최종 수정일
- 문서 버전 관리 (예: v1.0, v1.1)
- 관련 PRD 문서 링크

**예시**:
```
작성자: [이름/팀명]
작성일: 2024년 X월 X일
최종 수정일: 2024년 X월 X일
버전: 1.0
관련 PRD: [PRD 문서 링크]
```

---

### 2. 핵심 목표 (Goals)
**목적**: 이 제품/기능 개발을 통해 달성하고자 하는 구체적인 비즈니스 및 사용자 목표

**작성 가이드**:
- 비즈니스 목표와 사용자 목표를 구분하여 명시
- 구체적이고 측정 가능한 목표 설정
- 목표 달성 기간 명시 (선택 사항)

**예시**:
```
비즈니스 목표:
- 사용자 유입률 10% 증가 (3개월 내)
- 이탈률 5% 감소 (6개월 내)
- 매출 20% 증가 (연간)

사용자 목표:
- 사용자 작업 시간 30% 단축
- 사용자 만족도 4.5/5.0 달성
- 신규 사용자 온보딩 완료율 80% 이상
```

---

### 3. 성공 지표 (KPIs)
**목적**: 목표 달성 여부를 측정할 구체적인 지표

**작성 가이드**:
- **Success Metrics**: 긍정적인 목표 달성을 측정하는 지표
  - 각 목표에 대응하는 측정 가능한 지표 정의
  - 기준값(Baseline) 및 목표값(Target) 명시
- **Guardrail Metrics**: 부정적인 영향을 방지하고 모니터링하는 지표
  - 기능 출시로 인해 발생할 수 있는 부작용을 모니터링하는 지표
  - 예: 에러율 증가, 성능 저하, 사용자 이탈률 증가, 보안 이슈 등
  - 임계값(Threshold)을 명시하여 초과 시 즉시 대응할 수 있도록 설정
- 측정 주기 및 방법 간략히 설명

**예시**:
```
Success Metrics (성공 지표):
- MAU (Monthly Active Users): 현재 10,000 → 목표 15,000
- Retention Rate (7일): 현재 40% → 목표 50%
- Conversion Rate: 현재 5% → 목표 8%
- 평균 세션 시간: 현재 5분 → 목표 8분

Guardrail Metrics (안전장치 지표):
- 에러율: 현재 2% → 목표 0.5% 이하 (임계값: 1% 초과 시 알림)
- 사용자 이탈률: 현재 30% → 목표 25% 이하 (임계값: 35% 초과 시 알림)
- 평균 응답 시간: 현재 200ms → 목표 200ms 이하 (임계값: 500ms 초과 시 알림)
- 보안 이슈 발생 건수: 목표 0건 (임계값: 1건 이상 시 즉시 대응)

측정 방법:
- Zams 대시보드
- 주간 리포트 자동 생성
- Guardrail Metrics 임계값 초과 시 즉시 알림
```

### 4. 배경/동기 (Motivation)
**목적**: 이 기능/제품 개발을 시작하게 된 계기

**작성 가이드**:
- 시장 분석 결과 또는 트렌드
- 경쟁사 동향 및 차별화 포인트
- 내부 전략 및 비즈니스 요구사항
- 사용자 피드백 또는 요청사항

**예시**:
```
시장 동향:
- [관련 시장 트렌드 및 데이터]

경쟁사 분석:
- [경쟁사 A]: [주요 기능/차별점]
- [경쟁사 B]: [주요 기능/차별점]

내부 전략:
- [회사/팀의 전략적 방향성]

사용자 피드백:
- [주요 사용자 요청사항 또는 불만]
```

---

### 5. 사용자 및 페르소나 (User & Persona)
**목적**: 핵심 타겟 사용자 정의

**작성 가이드**:
- 주요 타겟 사용자 그룹 정의
- 페르소나 문서 링크 (있는 경우)
- 사용자 특성 및 니즈 요약

**예시**:
```
주요 타겟 사용자:
1. [사용자 그룹 A]: [특성 및 니즈]
2. [사용자 그룹 B]: [특성 및 니즈]

페르소나:
- 링크: [PERSONA_TEMPLATE.md 또는 페르소나 문서 링크]
- 대표 페르소나: [이름] - [한 줄 설명]
```

---

### 6. 해결할 문제 (The Problem)
**목적**: 사용자/비즈니스가 겪고 있는 핵심 문제를 명확하고 간결하게 정의

**작성 가이드**:
- 문제를 구체적이고 측정 가능하게 서술
- 문제의 영향 범위 및 심각도
- 문제 발생 빈도 또는 통계 데이터 (있는 경우)

**예시**:
```
핵심 문제:
현재 A 프로세스가 너무 복잡하여 70%의 사용자가 중도 포기함

문제 상세:
- 사용자가 [작업]을 완료하기까지 평균 10단계 필요
- 70%의 사용자가 5단계 이전에 이탈
- 사용자 만족도 조사에서 "복잡함"이 주요 불만 사항 (60%)

영향:
- 사용자 이탈률 증가
- 고객 지원 비용 증가
- 비즈니스 목표 달성 어려움
```

---

### 7. 현재 상태 (As-Is)
**목적**: (선택 사항) 현재 제품이나 기능이 작동하는 방식에 대한 간략 설명

**작성 가이드**:
- 현재 시스템/프로세스의 작동 방식
- 현재의 한계점 또는 문제점
- 기존 솔루션의 부족한 점

**예시**:
```
현재 프로세스:
1. 사용자가 [A] 화면에서 [작업] 시작
2. [B] 화면으로 이동하여 [정보] 입력
3. [C] 화면에서 [확인] 후 제출
...

현재의 한계:
- 화면 전환이 많아 사용자 혼란 유발
- 입력 정보가 분산되어 있어 실수 발생 가능
- 모바일 환경에서 사용성 저하
```

---

### 8. 솔루션 제안 (Proposed Solution)
**목적**: 정의된 문제를 어떻게 해결할 것인지에 대한 상위 수준의 설명

**작성 가이드**:
- 핵심 가치 제안 (Value Proposition)
- 솔루션의 주요 접근 방식
- 기대 효과
- 간결하고 명확하게 서술 (2-Pager 특성상 요약 중심)

**예시**:
```
핵심 가치 제안:
[문제]를 해결하기 위해 [솔루션]을 제공하여 [기대 효과]를 달성합니다.

주요 접근:
- [접근 방식 1]: [설명]
- [접근 방식 2]: [설명]
- [접근 방식 3]: [설명]

기대 효과:
- 사용자 작업 시간 50% 단축
- 사용자 만족도 향상
- 비즈니스 목표 달성
```
